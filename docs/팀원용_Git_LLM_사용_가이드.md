# 팀원용 Git + LLM 사용 가이드

우리 프로젝트에서 코드를 수정할 때는 아래 흐름만 꼭 지켜주세요.

---

## 1. 코드 수정 전에는 항상 최신 코드 받기

작업 시작 전에 먼저 내 로컬 코드가 최신인지 맞춰야 합니다.

```bash
git pull origin main
```

이걸 안 하고 코드를 고치면, 다른 팀원이 올린 코드랑 충돌이 날 수 있습니다.

흐름:

```text
GitHub 최신 코드
        ↓ pull
내 로컬 코드 최신화
        ↓
코드 수정 시작
```

LLM에게 시킬 때:

```text
코드 수정하기 전에 먼저 git pull origin main으로 최신 코드를 받아줘.
그 다음 [내가 수정하고 싶은 내용]을 작업해줘.
```

예시:

```text
코드 수정하기 전에 먼저 git pull origin main으로 최신 코드를 받아줘.
그 다음 크루 앱 홈 화면의 버튼 문구를 수정해줘.
```

---

## 2. 코드를 다 고쳤으면 commit & push 하기

코드를 수정하고 확인까지 끝났다면 GitHub에 올립니다.

```bash
git add .
git commit
git push origin main
```

LLM에게 시킬 때:

```text
수정한 내용을 확인하고 commit & push 해줘.
커밋 메시지에는 변경 내용, 변경 이유, 사용자 영향, 확인 사항을 포함해줘.
```

---

## 3. commit/push 할 때 적어야 하는 내용

push할 때 그냥 "수정함" 이렇게 쓰지 말고, 무엇을 고쳤고, 왜 고쳤고, 어떤 영향이 있는지 적어주세요.

형식:

```text
변경 내용
- 어떤 기능을 추가/수정했는지 적기
- 어떤 화면이나 파일이 바뀌었는지 적기

변경 이유
- 왜 이 수정이 필요했는지 적기
- 기존에 어떤 문제가 있었는지 적기

사용자 영향
- 사용자가 봤을 때 뭐가 좋아지는지 적기
- 앱 사용 흐름이 어떻게 바뀌는지 적기

확인 사항
- npm run build 통과
- 로컬 화면 확인 완료
- 버튼/화면 이동 확인 완료
```

예시:

```text
변경 내용
크루 앱 진행 화면에 실제 지도를 추가해 크루 현재 위치, 수거지, 허브를 함께 볼 수 있도록 수정했습니다.
크루 현재 좌표, 수거지 좌표, 허브 좌표와 거리 정보를 더 명확하게 표시하도록 보완했습니다.
홈 화면의 콜 목록을 수거 요청과 진행 중인 수거로 분리했습니다.

변경 이유
기존 목록 구조에서는 수락 전 요청과 진행 중 콜이 한 화면에 섞여 있어 흐름이 어색했습니다.
크루가 실제로 어디에 있는지, 수거지까지 얼마나 남았는지 한눈에 보기 어려웠습니다.

사용자 영향
크루원이 수락 후 진행 중인 수거를 별도로 관리할 수 있습니다.
진행 화면에서 현재 위치와 목적지, 거리 정보를 지도 기반으로 바로 확인할 수 있습니다.

확인 사항
npm run build 통과
진행 중인 콜 화면 렌더링 확인
수거 요청 / 진행 중인 수거 분리 구조 확인
```

LLM에게 시킬 때:

```text
이번 수정 내용을 바탕으로 커밋 메시지를 작성해줘.
형식은 변경 내용, 변경 이유, 사용자 영향, 확인 사항으로 정리해줘.
```

## 4. Fork한 사람은 Sync fork도 해야 함

내 GitHub에 fork한 저장소가 있다면, 원본 main 코드가 바뀔 때 내 fork에도 반영해야 합니다.

GitHub 화면에서:

```text
내 fork 저장소 들어가기
        ↓
Sync fork 버튼 클릭
        ↓
Update branch 클릭
```

LLM에게 물어볼 때:

```text
내 fork 저장소를 원본 main 최신 코드와 맞추고 싶어.
Sync fork를 어떻게 하면 되는지 단계별로 알려줘.
```

---

## 5. Contribute는 언제 쓰는 건가요?

내 fork에서 코드를 수정한 뒤 원본 프로젝트에 반영하고 싶을 때 사용합니다.

흐름:

```text
내 fork에서 코드 수정
        ↓
commit & push
        ↓
Contribute 버튼 클릭
        ↓
Open pull request
        ↓
팀원이 확인 후 Merge
```

즉, Contribute는 내가 수정한 코드를 원본 프로젝트에 반영 요청하는 기능입니다.

LLM에게 물어볼 때:

```text
내 fork에서 수정한 코드를 원본 저장소에 Pull Request로 올리고 싶어.
Contribute 버튼을 사용하는 방법을 단계별로 알려줘.
```

## 6. 전체 흐름도

일반 작업 흐름:

```text
[작업 시작]
   ↓
git pull origin main
   ↓
최신 코드 받은 상태에서 수정
   ↓
로컬에서 실행/확인
   ↓
git add .
   ↓
git commit
   ↓
변경 내용 / 변경 이유 / 사용자 영향 / 확인 사항 작성
   ↓
git push origin main
   ↓
GitHub 반영 완료
```

Fork를 쓰는 경우:

```text
원본 프로젝트 최신화 필요
   ↓
내 fork 저장소에서 Sync fork
   ↓
로컬에서 git pull origin main
   ↓
코드 수정
   ↓
commit & push
   ↓
Contribute
   ↓
Pull Request 생성
```

## 7. LLM에게 작업을 맡길 때 추천 프롬프트

코드 수정 시작할 때:

```text
작업 전에 git pull origin main으로 최신 코드를 먼저 받아줘.
그 다음 [수정하고 싶은 기능]을 구현해줘.
수정한 파일과 이유도 설명해줘.
```

화면 수정할 때:

```text
[앱 이름]의 [화면 이름]에서 [수정하고 싶은 부분]을 바꿔줘.
기존 디자인 흐름은 유지하고, 수정 후 어떤 파일을 바꿨는지 알려줘.
```

오류가 났을 때:

```text
이 오류가 왜 발생했는지 원인을 먼저 설명해줘.
바로 수정하지 말고, 어떤 파일을 확인해야 하는지 알려준 다음 수정해줘.
```

커밋/푸쉬할 때:

```text
수정한 내용을 git status로 확인하고,
문제 없으면 commit & push 해줘.
커밋 메시지는 변경 내용, 변경 이유, 사용자 영향, 확인 사항 형식으로 작성해줘.
```

충돌이 났을 때:

```text
git pull 또는 pull request 중 충돌이 났어.
충돌 파일을 보고 current change와 incoming change 중 어떤 걸 선택해야 하는지 설명해줘.
내 코드를 함부로 지우지 말고, 두 코드의 의도를 비교해서 해결해줘.
```

---

## 8. 최종 요약

```text
수정 전: git pull origin main
수정 후: commit & push
push할 때: 뭘 고쳤는지, 왜 고쳤는지 적기
fork 쓰면: Sync fork
원본에 반영 요청할 때: Contribute
LLM에게 시킬 때: pull 먼저, 수정 후 설명, push 전 변경 내용 정리 요청
```
